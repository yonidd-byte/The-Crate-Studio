# THE CRATE STUDIO — ENGINE ARCHITECTURE & ROADMAP
**Version:** 3.0 (Post Crate Brain & Sandbox UI Integration)
**Status:** Canonical reference for Engine Phasing and Sandbox Architecture. C++ Core Engine and Sandbox Architecture are FEATURE COMPLETE for V1 as of Phase 4, and hardened/profiled against a ruthless QA and performance-optimization pass in Phase 5.

## 0. Cross-Platform Strategy & macOS Porting
**Current Target:** Windows 10/11 (Win32/HWND) serves as the primary R&D environment and foundational proving ground. The low-level Windows APIs allow rapid iteration on extreme DSP manipulation, IPC, and GUI decoupling.
**macOS Directive:** The core C++ engine, `te::WaveNode` extraction, and IPC logic are inherently cross-platform. However, GUI Decoupling (HWND Reparenting) requires platform-specific implementations. Once the Windows version hits RC (Release Candidate) status and the engine is hermetically sealed, parallel macOS development will commence. The macOS port will translate Win32 auto-reset events to POSIX Condition Variables, and replace HWND Reparenting with Apple-compliant `NSView` bridging via XPC Services/App Sandbox rules.

## Phase 1: Core Isolation & GUI/DSP Decoupling (100% DONE)
*Objective: Run third-party code without risking the main audio thread or DAW stability.*

*   **IPC & Audio Bridge:** High-performance inter-process communication pipeline established.
*   **Shared Memory Sync:** Sample-accurate synchronization relying on OS-level shared memory to eliminate memory-copy overhead.
*   **Scatter-Gather & Yielding:** Proven stability running 50+ concurrent child processes, optimized with CPU yielding to prevent Spin-Wait core exhaustion.
*   **GUI/DSP Decoupling (HWND Reparenting):** Absolute segregation of rendering and processing. The DSP engine runs protected in the background, while the plugin's UI is successfully "stolen" (reparented) and embedded natively into the DAW's window structure.

## Phase 2: Nervous System, Time-Slip Engine & Immortality (100% DONE)
*Objective: Cheat death, manipulate time, and guarantee zero dropouts even during catastrophic plugin failure.*

*   **Continuous State Sync (Muscle Memory):** IPC backend listens for `endEdit` triggers. The VST Chunk (plugin state) is extracted continuously to the Parent DAW, creating a millisecond-accurate backup.
*   **Isolated Lookahead Extraction:** True "Alien Tech". A parallel `te::WaveNode` graph is constructed with an independent Playhead. It extracts raw audio blocks directly from the track's data structure at future timestamps, completely bypassing the live `te::Edit` state and preventing live-graph disturbance or plugin reinitialization.
*   **SSD Ring Buffer (Time-Slip):** Extracted future audio is cached in a lock-free Ring Buffer.
*   **The Watchdog (The Guillotine):** A strict 1000ms timeout on the Lookahead pipeline. If a third-party VST3 hangs, the Parent forcefully terminates the Child process instantly.
*   **Auto-Demotion & PluginHealthRegistry:** Following a Guillotine event, the DAW immediately fails over to the normal real-time Live Mode without audio dropouts. The misbehaving plugin is flagged as `unsafeForLookahead` and written to disk permanently. The DAW learns and never attempts Lookahead on this specific plugin again.

## Phase 3: The Crate Brain & Smart Confinement (100% DONE)
*Objective: Automated intelligence for resource management and system security.*

*   **PluginHealthRegistry (Profiling Database):** A local, persistent registry tracking `crashCount`, `unsafeForLookahead`, and `vendorName` per plugin, keyed by an authentic VST3 UID resolved by the Child (never re-scanned in the Parent, so untrusted factory code never executes inside the process the sandbox exists to protect). JSON-backed (`juce::DynamicObject`/`juce::JSON`, file at `%APPDATA%\The Crate Studio\CratePluginHealth.json`) rather than SQLite or a ValueTree XML store — this reuses and extends the registry already built in Phases 1-2 instead of standing up a second, parallel persistence layer for the same data.
*   **The Warden (`SandboxRouter`) — Optimistic Routing:** Pivoted away from the original vendor-whitelist policy — too punitive for RAM on a heavy project full of legitimate, unrecognized indie/boutique plugins. **Rule A — Shared Sandbox (The Fast Lane):** ANY plugin with `crashCount == 0` now routes here by default, regardless of vendor — "innocent until proven guilty," restored. **Rule B — Solitary Confinement:** a plugin is ONLY forced into isolation if it has actually crashed on this machine (`crashCount > 0`) or has been explicitly flagged `unsafeForLookahead` in the registry. Verified end-to-end: an unrecognized-vendor plugin with a clean crash history now correctly routes Shared instead of being blindly quarantined on first sight.
*   **Vendor Resolution IPC (Telemetry Only):** The Child still resolves `manufacturerName` at the same safe point pluginUID is resolved and still publishes it to the Parent over the Control Block — but under Optimistic Routing this data is no longer consulted by the Warden's routing decision at all. Retained purely for registry logging/telemetry (support diagnostics, a future "known vendors" view), never as a routing gate.
*   **Cryosleep Warm Pool (Zero-Footprint Pre-Warming):** A pool of pre-launched, idle `CrateSandbox.exe` processes waits fully booted but unclaimed, so a drag-and-drop load pays zero process-creation latency — no OS spawn overhead, no DLL-load stall, no visible stutter. An unclaimed process's heartbeat throttles from the normal 10ms active interval down to 1000ms while cryosleeping (~0.02% CPU noise per idle process, confirmed via direct measurement), and its `AudioBridgeThread`/`LookaheadWorkerThread` aren't even constructed until the moment it's actually claimed — the dominant idle cost was never the heartbeat itself, but these two threads existing at all. A claimed process is instantly backfilled via async replenishment so the pool stays at capacity without ever blocking the message thread.

## Phase 4: Sandbox UI Integration & Graphic Polish (100% DONE)
*Objective: Eliminate all visual evidence of the Sandbox. The hosted plugin must feel like a native application component.*

*   **Geometry Polish:** Complete eradication of "gray boxes" and margin artifacts via precise Client Area calculations against the host OS (`juce::HWNDComponent`'s pixel-precise `SetWindowPos`, WS_POPUP→WS_CHILD reparenting).
*   **Dynamic Resize (IPlugView) — Bidirectional Resize & Snap-Back:** The Child continuously republishes its editor's live width/height over the Control Block (not write-once); the Parent follows any size change — user drag, VST3-internal renegotiation, or scale-driven — the same way it already follows a post-crash handle change. No `setResizable()` toggling on a live desktop window (that call forces a full native-peer recreation and was the root cause of an earlier resize-stutter bug); content-driven auto-resize goes through plain `setSize()` instead. Host-initiated resizing (the user dragging the Parent's own resize corner) pushes the Parent's exact requested dimensions to the Child over IPC, applied through `ComponentBoundsConstrainer::setBoundsForComponent()` — not a plain `setBounds()`, which bypasses any attached constrainer entirely — the SAME public API JUCE's own corner-drag resizer uses, so `IPlugView::checkSizeConstraint()` genuinely gets to constrain the request to the nearest dimension the plugin actually supports. The Parent then snaps its own chrome back to whatever size the Child confirms was actually applied, even when that confirmed size never changes from the Child's perspective (a level-triggered check against the Parent's own current bounds, not just an edge-triggered "did the Child's value change" comparison — the latter alone misses a fixed-size plugin that always clamps back to the identical value). Resize IPC dispatch is debounced/throttled to at most once per 30ms during a continuous drag, with an exact final sync forced on mouse-up — preventing the flood of size renegotiations that was previously capable of freezing a plugin's own message loop under a fast, aggressive drag.
*   **Fixed-Size UX Lock:** The Child intercepts the VST3 `IPlugView::canResize()` capability at editor-creation time — read BEFORE the host wrapper's own `setResizable(true, true)` call, which unconditionally overwrites that flag for the Child's own internal resize plumbing — and publishes the plugin's real, honest answer to the Parent over the Control Block. If `canResize` is false, the Parent locks the host container and physically disables/hides the resize grip (`ResizableWindow::setResizable(false, false)`) the moment the true value is known, rather than leaving a fixed-size plugin's corner draggable only to snap back or unnecessarily trip the Editor View Recovery Guard.
*   **Workspace Bounds Clamping:** The Parent window is explicitly clamped every time a new size is applied — never larger than the current display's `Desktop::Displays::userBounds` (screen minus taskbar/dock), position included, so a plugin requesting an oversized UI can never bury its own resize grip off-screen. A `ComponentBoundsConstrainer` alone is insufficient here since content-driven resizing bypasses it entirely (only `setBoundsConstrained()` enforces one, and nothing in the resize-follow path calls that).
*   **DPI Bridge (4K & Multi-Monitor Scaling):** `juce::NativeScaleFactorNotifier` on the Parent's real top-level window detects genuine monitor-scale changes (the Child's own auto-detection freezes the instant its peer becomes a foreign-process WS_CHILD, per Microsoft's own documented cross-process DPI-broadcast limitations). The Parent pushes the corrected scale to the Child over the Control Block (`displayScale1000`, fixed-point); the Child applies it via `AudioProcessorEditor::setScaleFactor()` — JUCE's own VST3-hosting host-override hook straight into `IPlugViewContentScaleSupport`.
*   **Cross-Process Repaint Force:** JUCE's own embedded-HWND invalidation only reaches native Win32 child windows; a plain JUCE-rendered editor (no native child controls) was observed to go persistently blank after a fast resize burst outran the OS's default repaint. Fixed by explicitly forcing `InvalidateRect`/`UpdateWindow` on the raw embedded HWND after every applied size, independent of JUCE's own (insufficient, for this case) invalidation call.
*   **Editor View Recovery Guard (Self-Healing UI):** Two independent, complementary failure signals, not one — each catches a corruption the other cannot. (1) The Creation-Size Snap-Back fingerprint (a real, evidence-gathered third-party quirk observed with a MeldaProduction plugin): after a fast interactive resize burst, a plugin's own view can silently snap back to its exact creation-time size and then never report another change again — the Child fingerprints this exact signature (a hard revert to the precise original size, after having genuinely moved away from it) and recovers. (2) The HWND Responsiveness Ping: `SendMessageTimeoutW` with `SMTO_ABORTIFHUNG`, polled once a second against the editor's own native window, catches a message loop that has stopped pumping entirely — the failure mode observed under aggressive resize stress-testing (visual tearing, freeze, eventual crash) that the size-comparison fingerprint alone cannot see, since a genuinely frozen view may never report ANY size at all, let alone snap back to one. Either signal tears down and recreates just the corrupted editor view — same resilience philosophy as the Phase 2 Watchdog/Guillotine, scoped to the view alone since only the UI state is implicated, never the DSP/audio engine, which keeps playing throughout.

## Phase 5: The Debt Sweep — Hardening, Thread Safety & Performance Optimization (100% DONE)
*Objective: Before any further UI or Device Chain work, a ruthless QA/code-review and performance-profiling pass across the entire Sandbox and Anticipative FX engine — close every real correctness gap and cycle-waste found, not just the ones a feature request happened to surface along the way.*

*   **Kernel-Level Teardown (Zero-Zombies):** Every prior teardown path — `SandboxManager`'s own destructor, `CrateSandboxBridge::terminateSandboxProcess()`, `CrateSandboxApplication::shutdown()` — is a C++ destructor/explicit-call path, which runs on a clean exit only. A force-killed or crashed Parent process (Task Manager, an unhandled exception outside a clean unwind, a BSOD) skips every one of those, permanently orphaning any still-running `CrateSandbox.exe` — confirmed directly and repeatedly during development, where a manual `taskkill` was required after every hard-stop of the Parent. Fixed at the OS level, not the C++ level: a single, process-wide Win32 Job Object (`CreateJobObjectW` + `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`) is created once at startup, and every spawned `CrateSandbox.exe` — Cryosleep pool, Shared host, isolated/Solitary — is assigned to it the instant it launches via `AssignProcessToJobObject`. Since `juce::ChildProcess` never exposes the native `HANDLE`/PID a caller would need for that call, a minimal drop-in (`CrateIPC::JobObjectProcess`, matching `juce::ChildProcess`'s own `start()`/`isRunning()`/`kill()` surface) launches these processes directly via `CreateProcessW` instead. The result: Windows itself force-terminates the entire assigned process set the instant the Parent's handle table is torn down, on ANY exit, with zero reliance on our own cleanup code ever running — verified directly by force-killing the Parent mid-session and confirming zero orphaned `CrateSandbox.exe` processes survived.
*   **Real-Time Thread Safety — The StateExtractionThread Lock Fix:** A ruthless audit surfaced a genuine lock-safety bug, not a style nit: `StateExtractionThread`'s bounded spinlock-acquire loop (guarding `pluginAccessLock` against `AudioBridgeThread`'s own concurrent `processBlock()` call) called `plugin->getStateInformation()` and `pluginAccessLock.clear()` UNCONDITIONALLY, even when its 100-attempt backoff exhausted WITHOUT the lock ever actually being acquired. Under real contention this meant two compounding failures: a genuine unsynchronized data race calling into the SAME plugin instance while `processBlock()` was still touching it, and — worse — releasing a spinlock this thread never held, falsely handing out exclusive access to a second contender while the true holder was still mid-call, breaking mutual exclusion for every other thread relying on it. Fixed by tracking `acquired` explicitly and skipping both the plugin call and the clear() when the lock was never taken — the exact idiom `LookaheadWorkerThread` already had right for this same lock, now mirrored consistently across both consumers.
*   **Bounded Disk-I/O Thread Teardown:** `CrateAnticipativeWrapper`'s `ShadowWorker` performs genuine blocking disk I/O (the SSD overflow tier) on its own thread by design — a project folder on a network share or a spun-down external drive can legitimately make that I/O exceed the original 2000ms `stopThread()` budget. All three call sites (destructor, `initialise()`, `deinitialise()`) now route through a single `stopWorkerSafely()` helper: a first 2000ms attempt, and — only if that times out — a second, longer 5000ms attempt with a loud log, before proceeding. Destroying a `juce::Thread` whose `run()` hasn't actually returned is undefined behaviour; this closes that window without ever introducing an unbounded wait.
*   **Cache-Line Alignment (False Sharing Elimination):** A hardware-level audit of `CrateIPC::ControlBlock`'s memory layout found the audio round-trip handshake (`parentReady`/`childProcessed`/`numChannels`/`numSamples`) sharing a cache line with `heartbeatCounter` — a completely unrelated field written by an independent thread on its own ~10ms cadence. Cross-process shared memory does NOT exempt this from real MESI/MESIF cache-coherency ping-pong: the CPU's cache-coherency fabric operates on physical cache lines, not process boundaries, and this handshake is spin-waited on with `_mm_pause()` at audio-block frequency (1000+ times/sec per instance) — every unrelated heartbeat tick was invalidating the single hottest cache line in the system for zero benefit. Fixed with explicit `alignas(64)` separation: `heartbeatCounter` on its own line, the handshake group on another, `audioInput` on a third. The same different-writer false-sharing shape was found and fixed in the Lookahead ring's `lookaheadRequestHead`/`Tail` and `lookaheadResultHead`/`Tail` pairs (producer and consumer of each pair now on separate cache lines).
*   **Zero-Allocation Scrub:** `generateRealTrackAudioInto()` (the lookahead extraction path, driven by `LookaheadProducerThread`, which drains "as fast as the CPU allows" — a tight loop, not a throttled one) constructed a fresh `te::MidiMessageArray` on every single call. Hoisted to a persistent class member (`lookaheadScratchMidi`), cleared rather than reconstructed each call — consistent with the Pre-Allocated Memory Pool discipline already established everywhere else audio-adjacent in this codebase (Phase 1's Zero-Allocation Rule), extended here to a background thread that, while not hard-real-time, still runs hot enough that a per-call construction cost is pure waste.
